# PRD（プロダクト要件定義書） - エスカン

## 1. プロダクト概要

| 項目 | 内容 |
|------|------|
| プロダクト名 | エスカン |
| 種別 | SaaS型ウェブアプリケーション |
| ターゲット | SNS運用代行業者（代理店）およびそのクライアント企業 |
| 対象SNS | Instagram / TikTok（将来的にX(Twitter)も対応） |
| コアバリュー | 「スプレッドシートの柔軟性」と「専用ツールの管理能力」の融合 |

## 2. 解決する課題（3つのペイン）

### 2-1. 「確認・承認」の非同期コミュニケーションコスト
- **現状**: スプレッドシート、チャット、メール、共有フォルダが散乱し、最新版が不明になる
- **課題**: クライアントの確認待ちによる作業ストップ、修正履歴の管理ミス、「言った言わない」問題
- **解決策**: ブラウザ上での承認ボタン・修正コメントのピン留め機能による一元管理

### 2-2. 「レポート作成」の単純作業コスト
- **現状**: 各SNSの管理画面からCSVをダウンロードし、手作業でExcel/PowerPointに貼り付け
- **課題**: 分析（思考）ではなく作業（コピペ）に時間の8割を奪われ、利益率が低下
- **解決策**: Phase 2 でAPI連携による自動数値収集・テンプレート自動流し込み

### 2-3. 「属人的な知見」のブラックボックス化
- **現状**: 担当ディレクターの勘に頼った企画立案
- **課題**: 組織として「なぜ伸びたか」の再現性が確保しづらい
- **解決策**: 投稿データと成果指標の構造化による知見の蓄積・AI活用

## 3. ロール・権限設計

### 3-1. ロール一覧

| ロール | 説明 | 作成者 |
|--------|------|--------|
| マスター | システム全体の管理者。全企業を閲覧・管理できる | システム初期設定 |
| 企業管理者 | 1つの企業（代理店）の管理者。クライアント作成・スタッフ招待が可能 | マスター |
| スタッフ | 企業に紐付く作業者。複数企業に所属可能。クライアントのワークスペースに招待される | 企業管理者 / マスター |
| クライアント | 承認・閲覧を行う顧客。アカウント作成時は企業管理者配下にクライアントロールで参加 | 企業管理者が招待 |
| ゲスト | ログイン不要で共有リンク経由で閲覧のみ可能 | 企業管理者 / スタッフが発行 |

### 3-2. 権限マトリクス

| 機能 | マスター | 企業管理者 | スタッフ | クライアント | ゲスト |
|------|----------|------------|----------|-------------|--------|
| 企業の作成・管理 | ○ | × | × | × | × |
| クライアント作成・管理 | ○ | ○ | × | × | × |
| スタッフ招待 | ○ | ○ | × | × | × |
| ワークスペース閲覧 | ○ | ○ | ○（招待先のみ） | ○（自社のみ） | ○（リンク先のみ） |

| 投稿承認・差し戻し | ○ | ○ | × | ○ | × |
| 修正コメント | ○ | ○ | ○ | ○ | × |
| カレンダー閲覧 | ○ | ○ | ○ | ○ | ○ |
| AI機能利用 | ○ | ○ | ○ | × | × |
| ゲストリンク発行 | ○ | ○ | ○ | × | × |
| 分析・レポート閲覧 | ○ | ○ | ○ | ○ | ○（リンク先のみ） |

### 3-3. 組織構造

```
マスター
└── 企業（代理店）A
    ├── 企業管理者
    ├── スタッフ X（企業Aと企業Bに所属可）
    ├── クライアント（ワークスペース）1
    │   ├── スタッフ X（招待済み）
    │   ├── スタッフ Y（招待済み）
    │   └── クライアント担当者（承認者）
    └── クライアント（ワークスペース）2
        ├── スタッフ Z（招待済み）
        └── クライアント担当者（承認者）
```

## 4. フェーズ定義

### Phase 1（MVP）- 承認フロー管理 + カレンダー + 軽量AI

**目的**: コミュニケーションコストの削減（最も現場が苦しんでいる部分）

作るもの:
- 認証・ロール管理（マスター / 企業管理者 / スタッフ / クライアント / ゲスト）
- 企業・クライアント（ワークスペース）管理
- スタッフ招待・チーム管理
- 企画（Campaign）管理
- 投稿管理 CRUD + カレンダーUI
- 投稿の承認フロー（ステータス管理: 未着手 → 制作中 → 承認待ち → 修正中 → 予約済み → 投稿完了）
- SNS風プレビュー機能（Instagram / TikTok）
- 修正指示・コメント機能（対象秒数/ページ指定、ピン留め）
- 軽量AIキャプション下書き生成（OpenAI API）
- ゲスト閲覧リンク（ログイン不要）

### Phase 2 - 自動レポーティング

作るもの:
- Meta (Instagram) Graph API 連携
- TikTok API 連携
- 日次数値自動取得（フォロワー数、リーチ、保存数、プロフアクセス等）
- 自動計算指標（保存率、ホーム率、プロフ遷移率、フォロワー転換率等）
- ダッシュボード（リアルタイム数値確認）
- PDF レポート自動生成

### Phase 3 - AI企画・ライティングアシスタント本格版

作るもの:
- ペルソナ設定に基づくコンテンツ提案
- 過去の「伸びた投稿」分析によるパターン学習
- トーン＆マナーに合わせた複数パターンの提案
- ハッシュタグ最適化
- 投稿最適時間提案

## 5. Phase 1 コア機能詳細

### 5-1. クライアント共同編集・承認ダッシュボード

- カレンダー形式で投稿を一覧表示（月次/週次切替）
- 投稿カードにステータスバッジを表示
- ドラッグ＆ドロップで投稿日変更
- フィルタリング（ステータス別、担当者別、企画別）

### 5-2. 投稿管理

- **企画（Campaign）**: 「2月度運用」「求人シリーズ」等の括り
- **投稿データ（Post）**: 個別の投稿。1企画に複数投稿が紐付く
- 投稿種別: フィード / リール / ストーリーズ / TikTok
- 各投稿に含まれるデータ:
  - キャプション（テキスト）
  - ハッシュタグ
  - 素材（Google Drive URL）
  - 投稿予定日時
  - 担当者
  - ステータス
  - 承認履歴

### 5-3. プレビュー機能

- Instagram フィード風プレビュー（アイコン、ユーザー名、画像、キャプション表示）
- Instagram リール風プレビュー
- Instagram ストーリーズ風プレビュー
- TikTok風プレビュー
- Google Drive URL から画像/動画サムネイルを表示

### 5-4. 修正指示・コメント機能

- チャット形式のコメントスレッド
- メタデータ付き修正ログ:
  - 対象秒数/ページ（動画の○秒目、画像の○枚目）
  - 編集指示（具体的な修正内容）
  - 備考（補足事項）
  - ステータス（未対応 / 対応済み）

### 5-5. 承認フロー

- 承認ステップのカスタマイズ（企業単位で定義可能）
  - 例: 制作担当 → ディレクター → クライアント担当者 → クライアント決裁者
- ステップごとに「承認」「差し戻し」の操作
- 承認・差し戻し時のコメント入力
- ステータス自動遷移

### 5-6. 軽量AI機能（Phase 1）

- キャプション下書き生成（OpenAI API gpt-4o-mini）
- 入力: 投稿テーマ、ターゲット、トーン、参考テキスト
- 出力: キャプション案 3パターン
- ハッシュタグ候補の提案

### 5-7. ゲスト閲覧リンク

- ログイン不要でアクセス可能な共有URL発行
- 有効期限設定可能
- 閲覧範囲の制限（特定の企画/投稿のみ等）
- 閲覧専用（編集・承認不可）

## 6. 作らないもの（Phase 1 スコープ外）

- SNS API連携による自動投稿
- SNS API連携によるインサイトデータ自動取得
- 画像・動画の編集機能（Canva等の外部ツールに委ねる）
- 広告運用管理
- 炎上監視・アラート
- キャンペーン企画テンプレート
- 多言語対応（日本語のみ）
- ネイティブモバイルアプリ（レスポンシブWebで対応）

## 7. 非機能要件

### パフォーマンス
- ページ初期表示: 2秒以内
- カレンダー描画: 1秒以内
- API応答: 500ms以内

### セキュリティ
- Supabase Auth によるJWT認証
- Row Level Security (RLS) によるデータアクセス制御
- ゲストリンクはUUID + 有効期限で管理
- HTTPS必須

### 可用性
- Vercel + Supabase のマネージドインフラによる高可用性
- データベースの自動バックアップ（Supabase標準）

### スケーラビリティ
- 初期想定: 企業数 100社、クライアント数 500、同時接続 50ユーザー
- Supabase のスケーリングプランで対応
